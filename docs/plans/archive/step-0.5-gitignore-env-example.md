## Purpose

Two deliverables:

1. **`.gitignore` (edit existing file)** — the repo root `.gitignore` was auto-generated by Cargo scaffolding before the per-step discipline was formalized. It has useful entries that must be kept, but it is missing every spec item from ARCHITECTURE.md. This step audits and augments it: fix three trailing-slash violations, add eight missing spec items, reorganize with grouped section headers. The existing Rust-tooling entries stay.

2. **`.env.example` (create from scratch)** — a documentation file listing the four environment variables `kraxd` will eventually read. No code reads these variables yet; that arrives with `krax-config` in Phase 1+. This file documents the intended dev-environment configuration so a new contributor knows what to set.

These are the only files changed in this step (plus the standard ARCHITECTURE.md checkbox and AGENTS.md tracking update).

## Decisions resolved before this plan was written

1. **Existing `.gitignore` handling: audit-and-augment (option b).** The current file, in full:

   ```
   .claude

   # Generated by Cargo
   # will have compiled files and executables
   debug
   target

   # These are backup files generated by rustfmt
   **/*.rs.bk

   # MSVC Windows builds of rustc generate these, which store debugging information
   *.pdb

   # Generated by cargo mutants
   # Contains mutation testing data
   **/mutants.out*/

   # rustc will dump stack traces when hitting an internal compiler error to PWD
   rustc-ice-*.txt
   ```

   These entries are all legitimate (Claude Code project directory, Cargo debug/release output directories, rustfmt backups, MSVC debug symbols, mutants output, compiler ICE dumps). Replacing wholesale loses real coverage. The operation is a targeted edit, not a replacement.

2. **Three trailing-slash fixes required.** Current file has `target`, `debug`, and `.claude` without trailing slashes. All three are directories — confirmed via `ls -la`. Per git convention (Decision 3), directory entries get a trailing `/`. Fixed to `target/`, `debug/`, `.claude/`. The `.claude` case is the one the initial decision list missed; caught during plan sign-off.

3. **Grouped section headers.** Seven sections in the final file: `# --- Claude Code ---`, `# --- Build artifacts ---`, `# --- Dev data ---`, `# --- Secrets ---`, `# --- Editor ---`, `# --- OS ---`, `# --- Rust tooling ---`. The existing cargo-generated per-entry comments are dropped; the section header is sufficient.

4. **No net-new entries beyond the spec.** No `Thumbs.db` (Windows thumbnail cache), no `Cargo.lock`. Existing non-spec entries stay because they were already there; new additions require their own rationale PR. `Cargo.lock` stays committed — Krax is a binary project; Cargo's convention for binaries is to track the lock file.

5. **`KRAX_RPC_PORT` defaults to `9545`, not `8545`.** Port choices are sticky — every future script, README example, and integration test inherits the default. `9545` is the Optimism devnet convention and avoids the Anvil port conflict by default, producing a self-documenting devnet topology: kraxd on 9545, Anvil L1 on 8545.

6. **`.env.example` has one comment per variable** explaining purpose and valid values. `.env.example` is documentation, not just a key list.

7. **`.env.example` default values:** `KRAX_DATA_DIR=./data` (matches `make clean`'s `rm -rf data/`), `KRAX_RPC_PORT=9545`, `KRAX_L1_RPC_URL=http://localhost:8545` (Anvil default; comment notes Anvil must be running), `KRAX_LOG_LEVEL=info`.

8. **Documentation-only step.** No env-loading code, no `dotenvy`, `figment`, `config`, or any similar dependency added. Actual config loading arrives with `krax-config` in Phase 1+.

9. **LF line endings, trailing newline** on `.env.example`. Unix standard. Verified with a byte-level check in the Verification section.

10. **`git check-ignore` is the primary verification tool for `.gitignore`.** Tests what actually matters (does git interpret the entry correctly?) rather than approximating with grep. Per the Step 0.3 lesson: verification commands target what they actually verify.

### File 2 (CREATE): `.env.example`

**This is a new file — write from scratch at the project root.** LF line endings, trailing newline after the last line.

**Exact content:**

```
# Krax environment configuration.
# Copy this file to .env and fill in values for your environment.
# .env is gitignored; this file is documentation, not a live config.
# Actual config loading arrives with krax-config (Phase 1+).

# Directory where kraxd stores chain data (blocks, state).
# make clean removes this directory.
KRAX_DATA_DIR=./data

# Port for kraxd's JSON-RPC endpoint (eth_* and krax_* methods).
# Defaults to 9545 to avoid conflict with Anvil (L1 simulator) on 8545.
KRAX_RPC_PORT=9545

# HTTP URL of the L1 JSON-RPC endpoint (Anvil in dev, Sepolia/mainnet in prod).
# Run Anvil with: anvil (listens on 8545 by default).
KRAX_L1_RPC_URL=http://localhost:8545

# Log level for kraxd's tracing output.
# Valid values: error, info, debug, trace
KRAX_LOG_LEVEL=info
```

The file must end with a newline character (`0x0a`) after the last line. No blank line at the end beyond that required newline.

## Verification steps

Run in order from the project root. Every command must pass before the step is considered done.

```bash
# 1. Confirm each spec entry is present in .gitignore.
for entry in "target/" "debug/" "data/" "coverage/" ".env" ".env.local" ".idea/" ".vscode/" ".DS_Store" "*.log"; do
    grep -qF "$entry" .gitignore && echo "OK: $entry" || echo "MISSING: $entry"
done
# Expected: 10 "OK:" lines. Any "MISSING:" is a blocker.

# 2. Confirm the old unslashed directory entries are gone.
grep -E "^(target|debug|\.claude)$" .gitignore
# Expected: empty output. Any match here means a trailing-slash fix was missed.

# 3. Confirm .claude/ is present with trailing slash.
grep "^\.claude/$" .gitignore
# Expected: exactly ".claude/" — one line.

# 4. Confirm git check-ignore for three representative spec entries.
#    git check-ignore is the authoritative test: it asks git itself whether
#    the path would be ignored, which is the only thing that actually matters.
git check-ignore -v target/
git check-ignore -v data/
git check-ignore -v .env
# Expected: each command prints ".gitignore:N:<pattern>  <path>".
# Empty output means git does not consider the path ignored — a blocker.

# 5. Confirm git check-ignore for the retained non-spec .claude/ entry.
git check-ignore -v .claude/
# Expected: ".gitignore:N:.claude/  .claude/" — confirms the trailing-slash
# fix landed correctly.

# 6. Confirm Cargo.lock is NOT gitignored (binary-project convention).
git check-ignore -v Cargo.lock
# Expected: empty output. Cargo.lock is tracked; it must not be ignored.

# 7. Confirm .env.example exists and has exactly four KRAX_ variable lines.
grep -cE "^KRAX_" .env.example
# Expected: 4.

# 8. Confirm all four specific variable names are present with their defaults.
for line in "KRAX_DATA_DIR=./data" "KRAX_RPC_PORT=9545" "KRAX_L1_RPC_URL=http://localhost:8545" "KRAX_LOG_LEVEL=info"; do
    grep -qF "$line" .env.example && echo "OK: $line" || echo "MISSING: $line"
done
# Expected: 4 "OK:" lines.

# 9. Confirm .env.example has a trailing newline (last byte is 0x0a).
python3 -c "
import sys
data = open('.env.example', 'rb').read()
if data.endswith(b'\n'):
    print('OK: trailing newline')
    sys.exit(0)
else:
    print('FAIL: no trailing newline — last bytes: ' + repr(data[-4:]))
    sys.exit(1)
"
# Expected: "OK: trailing newline".

# 10. Confirm .env.example has no Windows line endings (no 0x0d bytes).
python3 -c "
import sys
data = open('.env.example', 'rb').read()
if b'\r' not in data:
    print('OK: LF only')
    sys.exit(0)
else:
    print('FAIL: CRLF detected')
    sys.exit(1)
"
# Expected: "OK: LF only".

# 11. Sanity check that .env itself would be ignored if created.
echo "TEST=1" > .env
git check-ignore -v .env
rm .env
# Expected: git reports .env as ignored. The rm cleans up the test file.

# 12. Confirm make lint still passes (no regressions — belt-and-suspenders).
make lint
echo "Exit code: $?"
# Expected: exit 0. .gitignore and .env.example changes cannot affect clippy,
# but running lint confirms the workspace is still clean.
```
